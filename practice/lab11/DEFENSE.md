# Lab 11 — Защита проекта (план ответа)

## 1. Общая идея проекта
- Это **ETL-пайплайн** (Extract-Transform-Load) — классический паттерн обработки данных
- Извлекаем данные из внешнего API (Random User Generator), преобразуем и загружаем в SQLite базу
- Пайплайн production-ready: есть логирование, обработка ошибок, инкрементальная загрузка, тесты

## 2. Extract (Извлечение данных)
**Что объяснять:**
- Используем библиотеку `requests` для HTTP-запроса к API `randomuser.me/api/?results=20`
- **Tenacity** — библиотека для retry-логики: если сеть упала, повторяем запрос до 3 раз с экспоненциальной задержкой (2с, 4с, 8с)
- **Инкрементальная загрузка**: передаём множество (set) уже загруженных email → фильтруем дубликаты ещё до трансформации
- API возвращает JSON с вложенными объектами (name, dob, location)

**Возможные вопросы:**
- *Зачем retry?* — Сеть ненадёжна, API может быть временно недоступен
- *Почему exponential backoff?* — Чтобы не перегружать сервер, каждый повтор ждём дольше

## 3. Transform (Преобразование)
**Что объяснять:**
- Разворачиваем вложенные JSON-поля: `name.first` → `first_name`, `dob.age` → `age`
- Добавляем вычисляемые колонки:
  - `age_group` — категоризация через `pd.cut()` (векторизованная операция, без циклов)
  - `email_domain` — доменная часть email через `str.split("@")`
  - `loaded_at` — метка времени загрузки (UTC)
- Удаляем дубликаты по email (`drop_duplicates`) и строки без email (`dropna`)
- **Всё через pandas-операции** (vectorized), без for-циклов — это быстрее

**Возможные вопросы:**
- *Что такое vectorized operations?* — Операции над всем столбцом сразу (как в NumPy), вместо поэлементного цикла
- *Зачем pd.cut вместо if/else?* — Работает на всём столбце сразу, O(n) без Python-цикла

## 4. Load (Загрузка)
**Что объяснять:**
- SQLite — встроенная в Python файловая БД, не требует сервера
- Таблица `users` с `email` как PRIMARY KEY
- `INSERT OR IGNORE` — если email уже есть, строка не дублируется (идемпотентность)
- Batch insert через `executemany` — одним запросом вставляем все строки

**Возможные вопросы:**
- *Почему INSERT OR IGNORE, а не UPSERT?* — Нам не нужно обновлять данные, только добавлять новые
- *Почему SQLite?* — Для учебного проекта достаточно, в продакшене можно заменить на PostgreSQL

## 5. Инкрементальная загрузка (Incremental Control)
**Что объяснять:**
- Таблица `etl_control` хранит состояние пайплайна (key-value)
- `get_existing_emails()` — получаем все email из БД → передаём в extract для фильтрации
- `update_last_email()` — сохраняем максимальный email (лексикографически) после каждого запуска
- Это позволяет **не загружать уже существующие данные** повторно

**Возможные вопросы:**
- *Зачем контрольная таблица?* — Чтобы пайплайн "помнил" что уже загружено между запусками
- *Зачем last_email?* — Для отслеживания последнего батча, можно использовать для аудита

## 6. Логирование
**Что объяснять:**
  - `RotatingFileHandler` — ротация логов: файл до 1 МБ, хранится 3 бэкапа
  - Уровни: INFO (начало/конец задач), WARNING (дубликаты), ERROR (исключения с traceback)
  - Лог пишется и в файл `etl_pipeline.log`, и в консоль

**Возможные вопросы:**
- *Зачем ротация?* — Чтобы логи не заняли весь диск в продакшене
- *Зачем разные уровни?* — Для фильтрации: в продакшене можно показывать только ERROR

## 7. Обработка ошибок и алертинг
**Что объяснять:**
- `run_etl()` обёрнут в try-except
- При ошибке: пишем в `alert.log` (имитация Slack-алерта) и перебрасываем исключение
- Скрипт завершается с ненулевым кодом → планировщик/CI увидит ошибку

## 8. Планировщик (scheduler.py)
**Что объяснять:**
- Отдельный скрипт, импортирует `run_etl`
- Запускает пайплайн 6 раз с интервалом 10 минут (1 час работы)
- Простой цикл с `time.sleep()` — подходит для демонстрации
- В продакшене использовали бы cron, Airflow или Prefect

## 9. Тесты (test_transform.py)
**Что объяснять:**
- 4 тестовых сценария через pytest:
  1. **Нормальные данные** — проверяем age_group, email_domain, first_name
  2. **Дубликаты** — остаётся только первая запись
  3. **Отсутствующий email** — строка удаляется
  4. **Пустой DataFrame** — возвращается пустой DataFrame с правильными колонками
- Используем хардкоженные DataFrames, без вызовов API
- Helper-функция `_make_user()` создаёт тестовые данные

**Возможные вопросы:**
- *Почему тестируем только transform?* — Extract зависит от API (нужен mock), Load от БД. Transform — чистая функция
- *Как запустить?* — `pytest test_transform.py -v`

## 10. Docker
**Что объяснять:**
- `Dockerfile`: базовый образ `python:3.11-slim`, копируем код, ставим зависимости
- Изоляция: не нужно ставить Python и зависимости на хост-машину
- `docker build -t etl-pipeline .` и `docker run --rm etl-pipeline`

## Ключевые термины для защиты
| Термин | Определение |
|---|---|
| ETL | Extract-Transform-Load — процесс извлечения, преобразования и загрузки данных |
| Incremental load | Загрузка только новых данных, а не всего набора заново |
| Exponential backoff | Стратегия повторных попыток с увеличивающейся задержкой |
| Idempotency | Повторный запуск даёт тот же результат (INSERT OR IGNORE) |
| Vectorized operations | Операции pandas на уровне массива без Python-циклов |
| Rotating log | Лог-файл с ограничением по размеру и автоматической ротацией |
