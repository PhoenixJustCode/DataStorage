# Шпаргалка перед уроком — Lab 12: Data Governance (SwiftPay)

---

## Что за задание вообще?

Проектируем систему управления данными для FinTech-стартапа SwiftPay (500к пользователей, мобильные платежи). Проблемы: три разных customer ID, грязные данные, никто не знает кто отвечает, регуляторы недовольны, ML-команда обучается на устаревших данных.

---

## Task 1 — Роли (Data Owner vs Data Steward)

**Data Owner** = бизнес-руководитель, принимает решения о данных, несёт ответственность  
**Data Steward** = специалист, следит за качеством на практике каждый день

| Домен | Data Owner | Data Steward |
|---|---|---|
| Customer (имя, email, адрес) | CPO / Product team | CRM-аналитик |
| Transactions (суммы, время) | CFO / Finance | Data Engineer |
| KYC (сканы документов) | CCO / Compliance | KYC-аналитик |

**Ключевой аргумент — почему НЕ IT:**
IT знает *как* хранить данные, но не знает *зачем* и что значит "правильные данные" в бизнес-смысле. Пример: IT скажет "адрес = непустая строка". Бизнес знает — адрес нужен для KYC, значит нужна страна + город + индекс. Это бизнес-решение.

## Task 2 — Две политики

**Структура политики:**
- **Purpose** — зачем она существует
- **Scope** — на какие данные распространяется
- **Rules** — конкретные требования
- **Consequences** — что будет при нарушении

**Политика 1: Качество email и адреса**
- Email верифицируется при регистрации + раз в 12 месяцев
- Адрес валидируется через внешний API (Google Address Validation) за 24 часа
- Дубли обнаруживаются автоматически → Data Steward закрывает за 5 рабочих дней
- Нарушение → план корректировки за 30 дней, эскалация в Governance Council

**Политика 2: Контроль доступа к чувствительным данным**
- Принцип минимальных привилегий (least privilege) — только то что нужно для работы
- Доступ к Restricted данным = одобрение Data Owner + срок до 90 дней + обоснование в IAM
- PAN-номера карт — никогда в открытом виде; ML-команда работает только с анонимизированными данными
- Квартальный ревью всех прав доступа, логи хранятся 12 месяцев
- Нарушение → блокировка аккаунта + расследование, утечка → уголовная ответственность + GDPR Article 33 (72 часа на уведомление регулятора)

## Task 3 — Операционная модель

**Три варианта:**

| Модель | Суть | Плюс | Минус |
|---|---|---|---|
| Централизованная | Одна команда решает всё | Единообразие | Узкое место, медленно |
| Децентрализованная | Каждый домен сам по себе | Гибкость | Хаос, несовместимость |
| **Федеративная** | Единые правила + локальная ответственность | Баланс | Сложнее координировать |

**Наш выбор: Федеративная (гибридная)**

Почему? SwiftPay не настолько большой, чтобы нужна была тяжёлая централизованная бюрократия. Но уже достаточно регулируемый, чтобы полная децентрализация вела к хаосу.

**Как решает проблемы:**
- Три разных customer ID → MDM + Business Glossary под контролем совета
- Непонятно кто отвечает → назначены Data Owners по доменам
- Регуляторы не видят lineage → централизованный каталог с lineage
- ML на устаревших данных → уведомления об изменениях source tables

**Data Governance Council:**
- CDO — председатель
- CPO / Product team — клиентские данные
- CFO / Finance — транзакционные данные
- CCO / Compliance — KYC данные
- Head of Data Science — потребитель данных
- Head of Engineering — техническое исполнение

## Task 4 — Три KPI

**KPI 1: Customer Data Completeness Rate**
```
(Записи без NULL в обязательных полях / Все активные клиенты) × 100%
```
Цель: ≥ 98% | Раз в неделю

**KPI 2: Data Lineage Coverage**
```
(Assets с задокументированным lineage / Все критические assets) × 100%
```
Цель: ≥ 95% для Tier-1 (регуляторная отчётность) | Раз в месяц

**KPI 3: Access Review Compliance Rate**
```
(Права доступа, прошедшие review в срок / Все права на review) × 100%
```
Цель: 100% — нулевая толерантность | Раз в квартал

## Task 5 — Data Catalog (proposal ~450 слов)

**Что такое Data Catalog?**
Wikipedia для данных внутри компании. Инженер открывает каталог и видит: где таблица, кто владелец, насколько качественные данные, откуда данные пришли (lineage), что означает каждое поле (glossary).

**Примеры инструментов:** DataHub (open-source), Collibra (enterprise), Alation

**Две ключевые фичи:**

1. **Automated Data Lineage** — автоматически отслеживает цепочку:
   ```
   Источник → ETL → Таблица → ML модель → Отчёт регулятору
   ```
   Compliance team показывает аудиторам полный путь данных.  
   Data Science team получает уведомления когда меняется upstream таблица → конец проблемы со stale data.

2. **Business Glossary + Data Certification** — единые определения:
   ```
   "Customer ID" в CRM        ──┐
   "customer_id" в транзакциях ──┤→ Glossary → одно определение → Certified dataset
   "user_id" в fraud системе  ──┘
   ```
   Устраняет проблему "три разных customer ID".

**Кто поддерживает каталог:**
- Data Stewards → обновляют метаданные, сертифицируют assets
- Data Engineers → настраивают автоматические коннекторы (crawlers, dbt)
- CDO office → управляет glossary, разрешает конфликты

**Главная проблема при внедрении: инженеры не хотят заполнять метаданные**

Решения:
1. Интеграция в IDE и Slack — убираем трение, не надо открывать отдельный инструмент
2. CI/CD gate — новая таблица без владельца не деплоится в production
3. Data Science team как первые пользователи → органические "евангелисты"

## Общая логика фреймворка (запомни на урок)

```
Governance Council (CDO + все владельцы доменов)
          ↓ устанавливает правила
    3 домена с Data Owner + Steward
          ↓ работают по правилам
    Data Catalog (lineage + glossary)
          ↓ даёт
    Single Source of Truth
```

**Главная мысль:** Governance — это не инструменты, а люди и процессы. Без Data Owners с реальной ответственностью любой каталог превратится в пустую базу метаданных.
